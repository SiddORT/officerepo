You are building a production-grade SaaS platform named "Office Repo".

Tech Stack:
- Frontend: React + TailwindCSS
- Backend: FastAPI
- Database: PostgreSQL
- Architecture: Multi-tenant SaaS (database-per-tenant)
- Authentication: JWT-based centralized identity system

IMPORTANT:
This project must follow strict enterprise coding standards, UI standards, validation standards, database standards and reusable architecture patterns from day one.

Backend structure:
- router layer
- service layer
- repository layer
- schema layer
- model layer

Frontend structure:
- reusable components
- centralized API client
- reusable form components
- reusable validation utilities

====================================================
4. DATABASE STANDARDS
----------------------------------------------------

Use PostgreSQL standards.

Rules:
- snake_case naming convention
- UUID support where needed
- created_at everywhere
- updated_at everywhere
- created_by where applicable
- soft delete support:
  is_deleted
  deleted_at

Use indexing where appropriate.

Use:
- foreign key constraints
- normalized schema design
- audit-ready structure

Never:
- duplicate tenant metadata in tenant DB
- store file blobs directly in DB

Store:
- file path only
- metadata only

====================================================
5. FILE UPLOAD / STORAGE STANDARDS
----------------------------------------------------

Create centralized file upload helper service.

IMPORTANT:
This helper must be reusable across all modules.

Future architecture:
- one S3 bucket per tenant

Current implementation:
- local storage only

Requirements:

Create helper:
shared/storage/

Capabilities:
- upload file
- generate unique filename
- create tenant folder automatically
- validate file size
- validate file extension
- return file path
- delete file
- replace existing file

Storage structure:

uploads/
   tenant_id/
      module_name/
         files

Store only:
- file path
- file metadata

inside database.

Do NOT store files directly in database.

Create abstraction layer so future migration to AWS S3 is easy.

Future-ready architecture:
- local storage strategy
- S3 storage strategy
